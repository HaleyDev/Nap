# Agent 使用规范 (Agent Conventions)

> 本文件是 AI Agent 在操作本项目时必须遵守的工作规范。
> 每一个新 Agent 会话在开始工作前，**必须先阅读本文件**。

---

## 1. 必读文件清单 (Required Reading)

Agent 在每次新会话开始时，**必须按顺序阅读**以下内容：

| 优先级 | 路径 | 说明 |
|--------|------|------|
| **P0** | `.agent.md` | 本文件 — Agent 工作规范 |
| **P0** | `docs/project-overview.md` | 项目详细介绍 — 快速掌握项目全貌 |
| **P0** | `docs/api-reference.md` | 接口文档 — 所有 API 端点的详细说明 |
| **P1** | `memory/` | 历史记录文件夹 — 阅读最近的会话记录，了解项目变更历史 |
| **P1** | `plans/` | 计划文件夹 — 阅读最新的计划文件，了解当前规划和已完成事项 |

### 1.1 读取 memory/ 历史记录

`memory/` 文件夹中存放着**历次 Agent 会话的工作记录**。文件命名格式为：

```
memory/YYYY-MM-DD-<简短描述>.md
```

例如：`memory/2026-04-23-vue3-migration.md`

Agent 应当：
- 阅读最近 **3~5 个**历史记录文件，了解近期变更上下文
- 如果要进行的修改与某个历史记录高度相关，应仔细阅读该记录
- **不要修改**已有的历史记录文件

### 1.2 读取 plans/ 计划文件

`plans/` 文件夹中存放着**项目的需求计划和开发规划**。

Agent 应当：
- 阅读 `plans/` 中的所有计划文件，了解项目的历史规划和当前需求
- 当收到新需求时，参考已有计划来做更好的规划
- 每次有**新的需求**时，在 `plans/` 中生成一份新的计划文件

计划文件命名格式：

```
plans/YYYY-MM-DD-<需求简述>.md
```

例如：`plans/2026-04-25-add-user-auth.md`

## 2. Agent 工作习惯 (Work Conventions)

### 2.1 开始工作前

1. **先读后写** — 在做任何修改前，先通读必读文件清单
2. **确认上下文** — 检查 memory/ 中是否有相关的历史工作记录
3. **参考计划** — 检查 plans/ 中是否已有相关的计划文件

### 2.2 工作过程中

1. **增量修改** — 优先修改已有文件，而非创建新文件
2. **保持风格一致** — 遵循项目已有的代码风格和目录结构
3. **测试验证** — 修改后确保项目能正常构建和运行
4. **及时记录** — 遇到重要决策或问题解决方案时，记录在心里以便写入历史记录

### 2.3 关键编码规范

- **后端**：Python 3.11+, FastAPI, async/await, SQLAlchemy 2.0 (Async), Pydantic V2
- **前端**：Vue 3 + TypeScript + Vuetify 3, Composition API (`<script setup lang="ts">`)
- **构建**：Vite 打包输出到 `app/templates/`，由 FastAPI 直接托管
- **部署**：Docker 多阶段构建 (Node.js 编译前端 → Python 运行后端)
- **数据库**：PostgreSQL，使用 asyncpg 驱动

### 2.4 目录结构约定

```
Nap/
├── .agent.md              ← Agent 规范（本文件）
├── docs/                  ← 项目文档
│   ├── project-overview.md
│   └── api-reference.md
├── memory/                ← Agent 会话历史记录
│   └── YYYY-MM-DD-xxx.md
├── plans/                 ← 需求计划文件
│   └── YYYY-MM-DD-xxx.md
├── frontend/              ← Vue 3 前端源码
│   └── src/
├── app/                   ← FastAPI 后端
│   ├── routers/
│   ├── services/
│   ├── static/uploads/
│   ├── templates/         ← Vue 构建输出（.gitignore）
│   ├── main.py
│   ├── models.py
│   ├── schemas.py
│   ├── config.py
│   └── database.py
├── Dockerfile
├── docker-compose.yml
└── requirements.txt
```

## 3. 会话结束后的历史记录生成 (Post-Session History)

在 Agent 会话结束后，用户可以**手动创建一份历史记录**保存到 `memory/` 文件夹中。

### 3.1 手动生成步骤

1. 在 `memory/` 文件夹下创建一个新的 Markdown 文件：

```bash
touch memory/YYYY-MM-DD-<简要描述>.md
```

2. 按以下模板填写内容：

```markdown
# 会话记录：<简要描述>

- **日期**：YYYY-MM-DD
- **Agent**：<使用的 Agent 名称/类型>
- **目标**：<本次会话要完成的任务>

## 完成的工作

- [ ] 工作项 1 — 简要说明
- [ ] 工作项 2 — 简要说明

## 修改的文件

| 文件路径 | 操作 | 说明 |
|----------|------|------|
| `path/to/file` | 新增/修改/删除 | 简要说明 |

## 遇到的问题与解决方案

### 问题 1：<问题标题>
- **原因**：...
- **解决**：...

## 关键决策

- 决策 1：为什么选择 A 而不是 B
- 决策 2：...

## 遗留事项

- [ ] 待办事项 1
- [ ] 待办事项 2
```

3. 保存文件，后续 Agent 会话会自动读取

### 3.2 让 Agent 自动生成

也可以在会话即将结束时，直接告诉 Agent：

> "请帮我生成一份本次会话的历史记录，保存到 memory/ 文件夹中"

Agent 将根据本次会话的工作内容自动生成记录文件。

## 4. 新需求时的计划生成 (Plan Generation)

> **核心原则：先问清楚，再动手写计划。计划的质量取决于需求的清晰度。**

### 4.1 计划编写的完整流程

当用户提出新需求时，Agent **严禁立即编写计划**，必须按以下流程执行：

#### 第一步：需求澄清（反复提问阶段）

用户提出需求后，Agent 必须进入 **需求澄清模式**，通过反复提问来消除所有不明确性。

**必须澄清的维度：**

| 维度 | 需要问清楚的问题 |
|------|------------------|
| **功能边界** | 这个功能具体做到什么程度？哪些是必须的，哪些是可选的？ |
| **用户场景** | 谁在什么场景下使用？有几种使用路径？ |
| **技术约束** | 有没有必须使用或必须避免的技术方案？ |
| **数据模型** | 需要新增哪些字段/表？现有数据结构是否需要改动？ |
| **UI/UX 期望** | 前端交互方式、页面布局、是否需要新路由？ |
| **异常处理** | 各种边界情况和错误场景怎么处理？ |
| **优先级** | 如果需求较大，哪些是第一期必须做的？ |
| **兼容性** | 是否需要向后兼容？对现有功能有什么影响？ |

**提问规范：**

- 每轮提问 **3~5 个问题**，不要一次性问太多
- 问题必须 **具体且有选项**，不要问开放性的模糊问题
- 用户回答后，根据答案 **继续追问衍生的不明确点**
- 直到用户确认 "可以开始了" 或 Agent 判断所有关键问题都已明确，才进入下一步

**示例对话：**

```
用户：我想加一个用户登录功能

Agent：好的，关于用户登录，我需要先确认几个关键点：
1. 认证方式：你倾向于 JWT token 还是 Session-based？考虑到现有的 FastAPI 架构，JWT 会更自然
2. 用户注册：需要支持自主注册，还是只由管理员创建账号？
3. 权限范围：登录后需要保护哪些功能？是所有 API 都需要认证，还是只有 Admin 相关的？

用户：JWT，不需要注册，只保护 admin

Agent：明白，那继续确认：
1. 管理员账号存储在哪里？数据库新建 users 表，还是直接写在 .env 配置里（单用户）？
2. 前端登录页面：需要独立的 /login 路由，还是在访问 /admin 时弹出登录对话框？
3. Token 过期策略：token 有效期多久？需要 refresh token 机制吗？

... （继续澄清直到所有细节明确）
```

#### 第二步：工具流程预研（如涉及外部工具/服务）

如果需求涉及 **对某个工具、第三方服务、或未使用过的技术的操作**，Agent 必须：

1. **先走通一遍完整流程** — 使用 Agent 实际操作，验证可行性
2. **记录踩坑点** — 把过程中遇到的问题和解决方案记录下来
3. **确认技术路径** — 基于实际验证结果，确定最终方案
4. **再编写计划** — 将验证过的方案写入计划，而非纸上谈兵

**触发条件**（符合任一即需预研）：

- 需要集成未使用过的第三方 API 或 SDK
- 需要使用新的数据库特性（如全文搜索、空间查询等）
- 需要配置新的基础设施组件（如 Redis、消息队列、S3 等）
- 涉及到不熟悉的协议或标准（如 OAuth2、WebSocket 等）
- 用户明确提到某个具体工具（如 "用 Alembic 做数据库迁移"）

**预研输出格式：**

```markdown
## 预研记录：<工具/技术名称>

### 验证步骤
1. 执行了 xxx 命令 → 结果：...
2. 调用了 xxx API → 结果：...

### 确认可行的方案
- 使用 xxx 方式，因为 ...

### 踩坑记录
- 问题 A：... → 解决：...
- 问题 B：... → 解决：...
```

#### 第三步：编写计划

在需求完全明确 + 技术路径验证通过后，才正式编写计划文件。

### 4.2 计划编写要求

1. **阅读已有计划** — 查看 `plans/` 中的历史计划，参考历史经验
2. **评估影响范围** — 分析新需求涉及的所有文件和模块
3. **生成新计划** — 在 `plans/` 中创建新的计划文件
4. **包含预研结论** — 如果做了工具预研，将结论纳入计划

### 4.3 计划文件模板

计划文件命名格式：`plans/YYYY-MM-DD-<需求简述>.md`

```markdown
# 计划：<需求标题>

- **日期**：YYYY-MM-DD
- **状态**：规划中 / 进行中 / 已完成
- **关联计划**：<如果有关联的历史计划，列出文件名>

## 需求描述

<详细描述用户需求 — 基于澄清阶段的所有讨论结论>

## 需求澄清记录

> 记录关键的澄清问答，便于后续回溯

| 问题 | 用户的回答 |
|------|------------|
| 认证方式？ | JWT |
| 用户存储？ | .env 单用户 |
| ... | ... |

## 预研结论（如有）

<记录工具/技术预研的关键结论和注意事项>

## 技术方案

### 方案概述
<整体技术方案描述>

### 涉及的文件修改
| 文件路径 | 修改类型 | 说明 |
|----------|----------|------|
| `path/to/file` | 新增/修改 | 简要说明 |

## 任务拆分

- [ ] Task 1：...
- [ ] Task 2：...

## 风险评估

- 风险 1：...
- 风险 2：...
```

### 4.4 总结：计划生成的黄金法则

```
❌ 用户提需求 → 立即写计划 → 开始编码
✅ 用户提需求 → 反复提问澄清 → (如需)工具预研验证 → 编写计划 → 用户确认 → 开始编码
```

**计划的质量 = 需求澄清的深度 × 技术验证的可靠性**

## 5. Git 提交规范 (Commit Convention)

### 5.1 Commit Message 格式

```
<type>(<scope>): <subject>

<body>
```

**第一行（必填）**：`<type>(<scope>): <subject>`

- **type** — 提交类型（必填）
- **scope** — 影响范围（可选，括号内）
- **subject** — 简要描述（必填，50 字符以内，中文或英文均可）

**空行 + body（选填）**：补充说明，可多行

### 5.2 Type 类型定义

| Type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(gacha): 添加关键字筛选抽取` |
| `fix` | 修复 Bug | `fix(api): 修复 f-string 引号语法错误` |
| `refactor` | 重构（不改变功能） | `refactor(frontend): 迁移前端到 Vue 3 + TS` |
| `style` | 代码格式调整（不影响逻辑） | `style: 统一缩进为 2 空格` |
| `docs` | 文档变更 | `docs: 添加 API 接口文档` |
| `chore` | 构建/工具/依赖等杂项 | `chore(docker): 改为多阶段构建` |
| `perf` | 性能优化 | `perf(db): 添加查询索引` |
| `test` | 测试相关 | `test(gacha): 添加保底机制单元测试` |
| `ci` | CI/CD 相关 | `ci: 添加 GitHub Actions 部署流程` |
| `revert` | 回滚提交 | `revert: 撤销 xxx 提交` |

### 5.3 Scope 范围定义

| Scope | 说明 |
|-------|------|
| `frontend` | 前端（Vue/TS/Vuetify） |
| `api` | 后端 API 路由 |
| `gacha` | 抽奖核心逻辑 |
| `db` | 数据库模型/迁移 |
| `docker` | Docker/部署相关 |
| `config` | 配置文件 |
| `upload` | 图片上传服务 |
| `agent` | Agent 规范/文档结构 |
| 无 scope | 跨模块或全局变更 |

### 5.4 Commit 示例

```bash
# 简单提交
git commit -m "feat(gacha): 添加 10 连抽保底机制"

# 带 body 的提交
git commit -m "fix(api): 修复 wish 接口分类为空时 500 错误

当 category 参数传空字符串时未正确转为 None，
导致数据库查询失败。现在会对空字符串做 strip + fallback 处理。"

# 多模块变更
git commit -m "refactor: 前端迁移到 Vue 3 + TypeScript + Vuetify 3

- 初始化 Vite + Vue 3 + TS 项目
- 实现 WishView 和 AdminView
- 添加 Vuetify 暗色主题配置
- 改写 FastAPI 为 SPA 托管模式
- Dockerfile 改为多阶段构建"
```

### 5.5 提交粒度原则

- **一个逻辑变更 = 一个 commit**，不要把多个不相关的修改混在一起
- **可编译** — 每个 commit 都应该是可编译/可运行的状态
- **大重构** — 拆分为多个有意义的 commit，而非一个巨型 commit
- **修复 + 功能分开** — bug fix 和 feature 不要混在同一个 commit
